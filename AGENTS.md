## Coding Conventions
- TypeScript server-only code stays in `apps/web/src/lib/server/` and uses `server-only` imports where applicable.
- Prefer Server Components; add `"use client"` only when necessary.
- Styling uses Tailwind v4 and established app patterns.

## API and Contract Rules
- TRR-Backend base comes from `TRR_API_URL` and is normalized to `/api/v1` in:
- `apps/web/src/lib/server/trr-api/backend.ts`
- Do not hardcode backend URLs.
- If backend response shape changes, update consumers in same session.

## Cross-Repo Implementation Order (Must Follow)
1. `TRR-Backend` contracts/schema first.
2. `screenalytics` consumers if impacted.
3. `TRR-APP` UI/integration updates.

## Validation and Handoff (Required)
During implementation and before ending session:
1. Run fast checks for changed app behavior.
2. Run targeted tests for changed routes/views.
3. Before any formal `<proposed_plan>` or documented multi-phase implementation plan, run `scripts/handoff-lifecycle.sh pre-plan`. This is not required for ad-hoc Q&A or one-off comments with no formal plan artifact.
4. If a task doc exists, update `docs/cross-collab/TASK*/STATUS.md` immediately after each completed implementation phase or materially completed plan step.
5. After each completed implementation phase when current state, blockers, or next action changed, update the canonical status source and then run `scripts/handoff-lifecycle.sh post-phase`.
6. Before ending the session or declaring PR-ready state, run `scripts/handoff-lifecycle.sh closeout`.
7. If `ACCEPTANCE_REPORT.md` exists, keep detailed validation evidence there instead of duplicating it into handoff.

`docs/ai/HANDOFF.md` is generated by `scripts/sync-handoffs.py`. Do not edit it by hand. Update canonical sources instead:
- `docs/cross-collab/TASK*/STATUS.md`
- `docs/ai/local-status/*.md`

Canonical sources that should surface in handoff must include a machine-parseable `## Handoff Snapshot` section:

````md
## Handoff Snapshot
```yaml
handoff:
  include: true
  state: active | blocked | recent | archived
  last_updated: YYYY-MM-DD
  current_phase: "<short phrase>"
  next_action: "<short phrase>"
  detail: self | "<relative/path.md>"
```
````

Generated `docs/ai/HANDOFF.md` output must stay short and use these sections:
- `Current Active Work`
- `Blocked / Waiting`
- `Recent Completions`
- `Archives / Canonical Links`

Detailed chronology belongs in `STATUS.md`. Detailed validation evidence belongs in `ACCEPTANCE_REPORT.md` when that file exists.
